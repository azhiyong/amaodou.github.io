dubbo笔记

发表于 2021-09-21

dubbo 版本 2.7.7

整体设计

  • config 配置层：对外配置接口，以 ServiceConfig, ReferenceConfig 为中心，可以直接初始化配置类，也可以通过 spring 解析配置生成配置类
  • proxy 服务代理层：服务接口透明代理，生成服务的客户端 Stub 和服务器端 Skeleton，以 ServiceProxy 为中心，扩展接口为 ProxyFactory
  • registry 注册中心层：封装服务地址的注册与发现，以服务 URL 为中心，扩展接口为 RegistryFactory, Registry, RegistryService
  • cluster 路由层：封装多个提供者的路由及负载均衡，并桥接注册中心（ Cluster 的目的是将多个 Invoker 伪装成一个 Invoker），以 Invoker 为中心，扩展接口为 Cluster, Directory, Router, LoadBalance
  • monitor 监控层：RPC 调用次数和调用时间监控，以 Statistics 为中心，扩展接口为 MonitorFactory, Monitor, MonitorService
  • protocol 远程调用层：封装 RPC 调用（封装了所有接口的透明化代理），以 Invocation, Result 为中心，扩展接口为 Protocol, Invoker, Exporter
  • exchange 信息交换层：封装请求响应模式，同步转异步，以 Request, Response 为中心，扩展接口为 Exchanger, ExchangeChannel, ExchangeClient, ExchangeServer
  • transport 网络传输层：抽象 mina 和 netty 为统一接口，以 Message 为中心，扩展接口为 Channel, Transporter, Client, Server, Codec
  • serialize 数据序列化层：可复用的一些工具，扩展接口为 Serialization, ObjectInput, ObjectOutput, ThreadPool

注入服务解析

org.apache.dubbo.config.spring.beans.factory.annotation.ReferenceAnnotationBeanPostProcessor

通过 spring bean 后置增强处理器，自动注入 dubbo 服务

本地 dubbo 服务直接返回 jdk 动态代理，远程服务使用 ReferenceBean#get 方法获取

org.apache.dubbo.config.ReferenceConfig#get 会调用 init 方法初始，根据服务消费者的配置构建远程 dubbo 服务的代理

org.apache.dubbo.config.ReferenceConfig#createProxy 根据指定协议创建服务执行器 invoker

返回服务执行器 invoker（默认是 FailoverClusterInvoker）的动态代理

org.apache.dubbo.rpc.protocol.ProtocolFilterWrapper#refer 协议包装类的 refer 方法，如果是 registry 协议，则调用 RegistryProtocol 的 refer 方法

org.apache.dubbo.registry.integration.RegistryProtocol#refer

创建 RegistryDirectory，返回 Cluster 创建的执行器 Invoker（FailoverClusterInvoker）

Cluster集群容错 https://dubbo.apache.org/zh/docs/advanced/fault-tolerent-strategy/

  • 这里的 Invoker 是 Provider 的一个可调用 Service 的抽象，Invoker 封装了 Provider 地址及 Service 接口信息
  • Directory 代表多个 Invoker，可以把它看成 List<Invoker> ，但与 List 不同的是，它的值可能是动态变化的，比如注册中心推送变更
  • Cluster 将 Directory 中的多个 Invoker 伪装成一个 Invoker，对上层透明，伪装过程包含了容错逻辑，调用失败后，重试另一个
  • Router 负责从多个 Invoker 中按路由规则选出子集，比如读写分离，应用隔离等
  • LoadBalance 负责从多个 Invoker 中选出具体的一个用于本次调用，选的过程包含了负载均衡算法，调用失败后，需要重选

SPI

获取扩展点加载器

org.apache.dubbo.common.extension.ExtensionLoader#getExtensionLoader 获取扩展点加载器，加载对应的实现类（创建 Class 对象），每个类型都对应一个扩展点加载器

ExtensionLoader 的构造函数先创建 ExtensionFactory 的扩展点加载器，objectFactory 是添加了 @Adaptive 注解的 ExtensionFactory (即 AdaptiveExtensionFactory）

org.apache.dubbo.common.extension.ExtensionFactory 接口只有1个方法

ExtensionFactory 扩展点有 3 个默认的实现

  • SpringExtensionFactory 依赖 Spring 容器，从 ApplicationContext 中获取指定的类型和名称的 Bean 实例

  • AdaptiveExtensionFactory 依赖其他的 ExtensionFactory 实现，调用依赖的 ExtensionFactory 获取扩展点

  • SpiExtensionFactory 通过扩展点加载器获取指定类型的扩展点实现

获取自适应的扩展点实例对象

org.apache.dubbo.common.extension.ExtensionLoader#getAdaptiveExtension 获取自适应的扩展实现

org.apache.dubbo.common.extension.ExtensionLoader#createAdaptiveExtension 获取自适应的扩展点实例，如果扩展点提供了标注@Adaptive的实现，则使用这个实现类的实例，没有提供@Adaptive注解的实现，则自动生成扩展点的实现代码，编译再加载到 jvm

org.apache.dubbo.common.extension.ExtensionLoader#getExtensionClasses 加载扩展点的实现类对象

从指定目录加载扩展点，dubbo 的 3 个扩展点目录

org.apache.dubbo.common.extension.ExtensionLoader#loadDirectory 获取扩展点的文件名称，通过类加载器加载资源

org.apache.dubbo.common.extension.ExtensionLoader#loadResource 解析扩展点文件内容，加载指定的实现类

org.apache.dubbo.common.extension.ExtensionLoader#loadClass 加载扩展点的具体实现类，通过4个缓存分别保存标注了@Adaptive注解的扩展点实现类、扩展点实现的包装类、标注了@Activate注解的扩展点实现、扩展点名称，默认情况下不能有同名的扩展点实现

根据名称获取扩展点实例对象

org.apache.dubbo.common.extension.ExtensionLoader#getExtension 根据指定名称获取扩展点实现

org.apache.dubbo.common.extension.ExtensionLoader#createExtension 获取扩展点的实现类，并完成实例化操作，如果扩展点有包装类，则实例化包装类（构造函数传入实际的扩展点实现），最终返回包装类对象

org.apache.dubbo.common.extension.ExtensionLoader#injectExtension 自动注入其他的扩展点实例对象
